### 1. SESSION KV binding is auto-injected with no id → `wrangler deploy` throws and is blocked
- **Severity:** HIGH · **Category:** bug · **Dimension:** Cloudflare Workers deployment correctness
- **Where:** `apps/web/wrangler.jsonc:14-16`
- **Problem:** wrangler.jsonc declares only the SITES KV namespace. Astro sessions are ON by default and astro.config.ts does not set `session: false`, so the @astrojs/cloudflare adapter (v13.6.0) silently injects a second KV binding `{ binding: 'SESSION' }` WITH NO id into the merged deploy config. A KV binding without an id makes `wrangler deploy` fail. ROADMAP line 13 claims SESSION is configured in wrangler.jsonc and verified ('build → Worker; merged deploy config'), and the operator checklist (ROADMAP:54) only says `wrangler kv namespace create SITES` — SESSION is never declared and never provisioned, so the very first real `pnpm deploy` aborts.
- **Fix:** Primary fix (provably correct): declare a real SESSION KV namespace in apps/web/wrangler.jsonc so the adapter suppresses its idless auto-injection. Per the adapter's dist/wrangler.js:11-16, when the user config already contains a kv_namespaces entry whose binding === "SESSION", `hasSessionBinding` is true and the adapter emits `void 0` (no injection). Add to the top-level `kv_namespaces`: `{ "binding": "SESSION", "id": "<real session kv id>", "preview_id": "<real session preview kv id>" }`, and redeclare it inside the `[env.staging]` kv_namespaces block (env blocks do not inherit top-level bindings — already noted in the file's own comment at lines 32-33). Then add `wrangler kv namespace create SESSION` to the provisioning checklist (ROADMAP line 54) alongside the existing SITES step. Also fix the false ROADMAP claim at line 13 — SESSION is not actually in wrangler.jsonc. Do NOT rely on the finding's alternative option (b) `session: false` in astro.config.ts: it does not work with this adapter. The guard `if (!session?.driver)` (dist/index.js:88) still evaluates true when `session` is `false` (because `false?.driver` is undefined), so the adapter re-creates the KV session driver and injects the idless SESSION binding regardless. If sessions are genuinely unused and you still want to avoid provisioning a namespace, do not assume `session: false` suffices — verify the emitted dist/server/wrangler.json no longer contains an idless SESSION entry, and prefer overriding the binding to one you do declare (the documented `sessionKVBindingName` adapter option pointing at a real binding) rather than trying to disable it. The reliable, code-proven fix remains declaring a real SESSION namespace. Validation: after the fix, run the deploy and confirm; or run `wrangler deploy --dry-run --outdir /tmp/x` against the merged config to confirm no "SESSION bindings must have an \"id\" field" error (note dry-run alone is insufficient because cli.js:168094 bypasses the id check — a real deploy or inspection of the merged dist/server/wrangler.json kv_namespaces is the true check).

### 2. rss.xml emits the build-time static origin, not the per-host registry canonicalUrl
- **Severity:** HIGH · **Category:** contract-drift · **Dimension:** Multidomain host-resolution edge cases
- **Where:** `apps/web/src/pages/rss.xml.ts:7-13`
- **Problem:** ROADMAP Phase 2 claims canonical is sourced 'from registry' for every emitted URL, and the sibling endpoints sitemap.xml.ts (line 10) and robots.txt.ts (line 14) both correctly use `locals.site.canonicalUrl`. The RSS endpoint instead falls back to `context.site ?? SITE.url`. In an Astro SSR endpoint `context.site` is the build-time `site` from astro.config.ts (a single static `process.env.SITE_URL ?? 'https://blog.etus.example'`), which never varies by Host. So on a multidomain deployment EVERY tenant's /rss.xml advertises the build-time placeholder origin (blog.etus.example) as `<link>`/channel origin, and the item `link: /posts/${id}/` values resolve against that wrong origin. The feed is reachable on every host (linked globally from Nav.astro `{ href: '/rss.xml', label: 'RSS' }`). This is a direct canonical-consistency violation: feed-discovered URLs split ranking/attribution to the wrong domain.
- **Fix:** Convert the endpoint to `export const GET: APIRoute = async ({ locals }) => {...}` and derive the origin from the Host-resolved registry, mirroring sitemap.xml.ts/robots.txt.ts: `const origin = locals.site.canonicalUrl.replace(/\/+$/, '')`, then pass `site: origin` to rss(). Passing the correct `site` alone already fixes the item links too, because @astrojs/rss resolves relative item `link` values against `site` via `new URL(link, site)` (dist/index.js:143-144); making them absolute (`link: `${origin}/posts/${p.id}/``) is optional belt-and-suspenders. Also replace the static `SITE.brand`/`SITE.description` with the per-tenant `locals.site.brand`/`locals.site.description` so the feed title/description match the resolved domain. Never use `context.site` or `SITE.url` for a per-host feed.

### 3. X-Forwarded-Host trusted above the real Host enables cross-tenant config resolution + URL poisoning
- **Severity:** HIGH · **Category:** security · **Dimension:** Multidomain host-resolution edge cases
- **Where:** `apps/web/src/middleware.ts:18-19`
- **Problem:** Host resolution prefers the `x-forwarded-host` request header over the real `host`. Cloudflare does NOT strip an inbound client-supplied X-Forwarded-Host (it is not part of the trusted CF-* header set; CF only manages X-Forwarded-For/CF-Connecting-IP/cf.* and the real Host). Any client can send `X-Forwarded-Host: othertenant.com` to a tenant's edge and make resolveSite/KV select a DIFFERENT site's SiteConfig than the domain actually being served. Because every per-host endpoint derives its output from the resolved `locals.site` (ads.txt seller lines + OWNERDOMAIN/MANAGERDOMAIN, sellers.json, robots Sitemap:, sitemap loc origin, and the canonical/og:url origin via site.canonicalUrl), an attacker controls which tenant's config (and which canonical origin) is emitted on a victim host — the exact 'attacker-controlled Host header poisons emitted URLs' risk for this dimension. XFH being checked FIRST makes it strictly worse than trusting Host alone.
- **Fix:** On the Cloudflare Workers runtime the Worker is the edge, so resolve identity from the real Host only: `const host = req.headers.get('host') ?? new URL(req.url).host`. Drop `x-forwarded-host` from the precedence chain entirely (middleware.ts:18-19) — do not consult it unless it arrives from a trusted reverse proxy you operate AND you strip/overwrite any client-supplied value before reading it; if a known proxy genuinely needs it, gate it behind an allowlist/shared-secret check rather than blind first-precedence. As defense-in-depth on the per-host endpoints (ads.txt/sellers.json/robots/sitemap), either remove `cache-control: public` or add `Vary: X-Forwarded-Host` (better: stop honoring XFH so the cache key is the real Host) to prevent cache-poisoning of the resolved-config/canonical-origin output to other users. This becomes exploitable for true cross-tenant config disclosure the moment 2+ sites exist in SITES KV / the registry, so fix before multi-tenant go-live.

### 4. Stored XSS: tenant-controlled ad config breaks out of inline <script> via unescaped JSON.stringify
- **Severity:** HIGH · **Category:** security · **Dimension:** Security — inline-script XSS, secrets, headers, tenancy
- **Where:** `packages/ads/src/emit.ts:82-91`
- **Problem:** Every inline emitter interpolates attacker-influenceable values (adUnit path, targeting keys+values, networkCode, secureSignalProviders, pps, interstitial triggers, refresh) into a JS string using JSON.stringify(...), and the result is rendered via `<script is:inline set:html={script}>` in AdSlot/AdAnchor/AdSideRail/AdInterstitial/AdScripts. JSON.stringify does NOT escape `<`, `>`, or `/`, so it does NOT neutralize the byte sequence `</script>`. In a multi-tenant system the SiteConfig (ads.networkCode, Placements.adUnit, Placements targeting/sizes) is authored by the tenant in Payload (Sites/Placements collections), mirrored host->SiteConfig into the SITES KV, and read by middleware into Astro.locals.site with no validation/escaping. A tenant who sets adUnit or a targeting value to contain `</script>` terminates the inline script element at the HTML-parser level and injects arbitrary markup that runs in the site's origin (cookies, localStorage incl. the PPID, DOM, and any same-origin endpoints). This is a real, exploitable injection surface, not theoretical.
- **Fix:** Do not rely on JSON.stringify for an inline <script> (HTML) context — it does not neutralize </script>/<!--. Fix at the serialization boundary in emit.ts: HTML-escape the serialized JSON before it reaches set:html by replacing `<`→<, `>`→>, `/`→/, `&`→&, and U+2028/U+2029 (e.g. via `serialize-javascript` or a small jsesc-style helper), OR stop emitting tenant data into JS strings entirely and pass it via data-* attributes read by a static script — serialize.ts already implements exactly this pattern (slotDataAttrs/oopDataAttrs) for the client-bundle path, so the safe primitive exists. Apply to ALL emitters: bootSlotScript (adUnit/targeting/sizes/refresh/id), bootGptScript (secureSignalProviders/pps/privacy/ppid), bootInterstitialScript + oop (path/triggers), and the JSON-LD sink at SeoHead.astro:31 (same class — tenant/content-authored fields). As defense-in-depth, validate networkCode and adUnit against a strict charset at the CMS boundary (cms/collections.ts:38,118 — add a validate() limiting to digits / [A-Za-z0-9_/-]) and ideally re-validate when reading from KV in middleware.ts. Track this as a hard prerequisite (DoD) for Phase 6 go-live, since that is when the untrusted CMS→KV data path becomes operational and this escalates to critical.

### 5. Stored XSS: JSON-LD sink uses unescaped JSON.stringify, allowing </script> breakout from brand/description/author
- **Severity:** HIGH · **Category:** security · **Dimension:** Security — inline-script XSS, secrets, headers, tenancy
- **Where:** `packages/seo/src/components/SeoHead.astro:31`
- **Problem:** The JSON-LD block is emitted with `<script is:inline type="application/ld+json" set:html={JSON.stringify(cleanJsonLd)}>`. The JSON-LD object is built from SiteMeta (site.brand) and SeoProps (title, description, author.name/url) which originate from the tenant-authored SiteConfig (brand, description) and Posts/Authors content. Same root cause as the ads emitters: JSON.stringify does not escape `</script>`, so a brand/description/author value containing `</script><script>...</script>` breaks out of the ld+json script element and executes. Even inside a type=application/ld+json script the HTML parser still honors `</script>` as the element terminator, so this is exploitable XSS, not inert text.
- **Fix:** Escape the serialized JSON before set:html, but use JSON-string-safe Unicode escaping, NOT HTML-entity encoding. Inside a <script> element the content is not HTML-parsed for entities, so the original suggestion to 'replace < with &lt;' is wrong — &lt; would be emitted literally and corrupt the JSON (the parser does not decode entities in script data). The correct fix: after JSON.stringify, replace `<`→`<`, `>`→`>`, `&`→`&`, U+2028→` `, U+2029→` ` (these backslash-u sequences are valid inside JSON strings and neutralise `</script>`, `<!--`, and the U+2028/U+2029 line terminators that break inline JS). Example: const safe = JSON.stringify(cleanJsonLd).replace(/</g,'\\u003c').replace(/>/g,'\\u003e').replace(/&/g,'\\u0026').replace(/ /g,'\\u2028').replace(/ /g,'\\u2029'); then set:html={safe}. Factor this into one shared safe inline-JSON serializer and reuse it for the @etus/ads emitters (packages/ads/src/emit.ts interpolates many unescaped JSON.stringify(...) values into inline <script> strings — e.g. lines 55, 63, 66, 80, 83, 86, 89, 91 — and has the identical breakout exposure for any attacker-influenced ad config/targeting). Adding a strict Content-Security-Policy (Phase 12) is worthwhile defense-in-depth but does not substitute for fixing the escaping, and must not be relied on alone.

### 12. Middleware trusts client-supplied x-forwarded-host over the real Host, allowing site-config spoofing / cache poisoning
- **Severity:** MEDIUM · **Category:** security · **Dimension:** Astro SSR runtime + dynamic endpoints (HTTP hygiene)
- **Where:** `apps/web/src/middleware.ts:18-19`
- **Problem:** Host resolution prefers the client-controlled `x-forwarded-host` header ABOVE the genuine `host` header. On Cloudflare Workers nothing upstream sets `x-forwarded-host`, so it is fully attacker-controlled. A request to siteA with `x-forwarded-host: siteB.com` resolves the SiteConfig for siteB — serving siteB's ads.txt seller IDs / MANAGERDOMAIN, canonical origin, robots sitemap origin and theme on siteA's domain. Because every dynamic endpoint sets `cache-control: public, max-age=3600` and the Cloudflare edge cache key does not include arbitrary request headers like `x-forwarded-host`, a single forged request can poison the shared cache so other visitors of siteA receive siteB's ads.txt/robots/canonical (cache-poisoning). This defeats the documented 'fail-closed, never a wrong site's config' guarantee.
- **Fix:** Stop trusting `x-forwarded-host` on the public edge. Derive host from `req.headers.get('host') ?? new URL(req.url).host` only (middleware.ts:18-19). On Cloudflare Workers with no trusted reverse proxy, `x-forwarded-host` is client-controlled and must not influence SiteConfig resolution. If a trusted proxy is ever introduced, gate `x-forwarded-host` behind a verified shared-secret/allowlist check before honoring it. Since the header dependency should be removed entirely, a `Vary: X-Forwarded-Host` header is the wrong fix and is unnecessary. Separately (defense-in-depth), reconsider `cache-control: public, max-age=3600` on these host-varying endpoints: because the body varies by Host, a `public` cached copy in any shared downstream proxy could be served for the wrong domain; once host trust is fixed the body correctly varies only by the real Host, but keeping these responses `private`/short-TTL or ensuring any caching keys on Host avoids cross-site bleed.
